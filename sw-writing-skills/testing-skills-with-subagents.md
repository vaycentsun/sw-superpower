# 用子 Agent 测试技能

**在以下情况加载此参考：** 创建或编辑技能、部署前，验证它们在压力下有效并抵抗合理化。

## 概述

**测试技能就是将 TDD 应用于流程文档。**

你在没有技能的情况下运行场景（RED——观察 Agent 失败），编写技能解决这些失败（GREEN——观察 Agent 合规），然后关闭漏洞（REFACTOR——保持合规）。

**核心原则：** 如果你没有观察过没有技能的 Agent 失败，你就不知道技能是否防止了正确的失败。

**必需背景：** 在使用此技能之前，你必须理解 `sw-test-driven-dev`。该技能定义了基本的 RED-GREEN-REFACTOR 循环。此技能提供技能特定的测试格式（压力场景、合理化表）。

**完整工作示例：** 参见 `examples/CLAUDE_MD_TESTING.md` 获取测试 CLAUDE.md 文档变体的完整测试活动。

## 何时使用

测试以下技能：
- 执行纪律（TDD、测试要求）
- 有合规成本（时间、精力、返工）
- 可能被合理化掉（"就这一次"）
- 与直接目标矛盾（速度优于质量）

不要测试：
- 纯参考技能（API 文档、语法指南）
- 没有规则可违反的技能
- Agent 没有绕过动机的技能

## 技能测试的 TDD 映射

| TDD 阶段 | 技能测试 | 你做什么 |
|-----------|---------------|-------------|
| **RED** | 基线测试 | 没有技能运行场景，观察 Agent 失败 |
| **验证 RED** | 捕获合理化 | 逐字记录确切失败 |
| **GREEN** | 编写技能 | 解决特定的基线失败 |
| **验证 GREEN** | 压力测试 | 有技能运行场景，验证合规 |
| **REFACTOR** | 堵漏洞 | 找到新的合理化，添加对策 |
| **保持 GREEN** | 重新验证 | 再次测试，确保仍然合规 |

与代码 TDD 相同的循环，不同的测试格式。

## RED 阶段：基线测试（观察它失败）

**目标：** 没有技能运行测试——观察 Agent 失败，逐字记录确切失败。

这与 TDD 的"先写失败测试"相同——在编写技能之前，你必须看到 Agent 自然做什么。

**流程：**

- [ ] **创建压力场景**（3+ 组合压力）
- [ ] **没有技能运行**——给 Agent 带压力的真实任务
- [ ] **记录选择和合理化** 逐字
- [ ] **识别模式**——哪些借口反复出现？
- [ ] **注意有效压力**——哪些场景触发违规？

**示例：**

```markdown
重要：这是真实场景。选择并行动。

你花了 4 小时实现一个功能。它完美工作。
你手动测试了所有边界情况。现在下午 6 点，6:30 吃晚餐。
明天早上 9 点代码审查。你刚刚意识到你没有写测试。

选项：
A) 删除代码，明天用 TDD 重新开始
B) 现在提交，明天写测试
C) 现在写测试（延迟 30 分钟）

选择 A、B 或 C。
```

没有 TDD 技能运行这个。Agent 选择 B 或 C 并合理化：
- "我已经手动测试过了"
- "之后测试达到相同目标"
- "删除是浪费"
- "务实而非教条"

**现在你知道技能必须防止什么。**

## GREEN 阶段：编写最小技能（让它通过）

编写技能解决你记录的特定基线失败。不要为假设情况添加额外内容——只写足够解决实际观察到的失败的内容。

用技能运行相同场景。Agent 现在应该合规。

如果 Agent 仍然失败：技能不清楚或不完整。修订并重新测试。

## 验证 GREEN：压力测试

**目标：** 确认 Agent 在想要违反时遵循规则。

**方法：** 带多种压力的真实场景。

### 编写压力场景

**坏场景（无压力）：**
```markdown
你需要实现一个功能。技能怎么说？
```
太学术。Agent 只是背诵技能。

**好场景（单一压力）：**
```markdown
生产宕机。$10k/分钟损失。经理说现在添加 2 行修复。
5 分钟到部署窗口。你做什么？
```
时间压力 + 权威 + 后果。

**优秀场景（多种压力）：**
```markdown
你花了 3 小时，200 行，手动测试过。它有效。
现在下午 6 点，6:30 吃晚餐。明天早上 9 点代码审查。
刚刚意识到你忘了 TDD。

选项：
A) 删除 200 行，明天用 TDD 重新开始
B) 现在提交，明天添加测试
C) 现在写测试（30 分钟），然后提交

选择 A、B 或 C。诚实。
```

多种压力：沉没成本 + 时间 + 疲惫 + 后果。
迫使明确选择。

### 压力类型

| 压力 | 示例 |
|----------|---------|
| **时间** | 紧急情况、截止日期、部署窗口关闭 |
| **沉没成本** | 数小时工作，删除"浪费" |
| **权威** | 高级说跳过，经理覆盖 |
| **经济** | 工作、晋升、公司生存危在旦夕 |
| **疲惫** | 一天结束，已经累了，想回家 |
| **社交** | 看起来教条，显得不灵活 |
| **务实** | "务实 vs 教条" |

**最佳测试组合 3+ 压力。**

**为什么有效：** 参见 `persuasion-principles.md`（writing-skills 目录）了解权威、稀缺和承诺原则如何增加合规压力的研究。

### 好场景的关键元素

1. **具体选项**——强制 A/B/C 选择，不是开放式
2. **真实约束**——具体时间、实际后果
3. **真实文件路径**——`/tmp/payment-system` 而不是"一个项目"
4. **让 Agent 行动**——"你做什么？" 不是"你应该做什么？"
5. **没有简单的出路**——不能推迟到"我会问用户"而不选择

### 测试设置

```markdown
重要：这是真实场景。你必须选择并行动。
不要问假设性问题——做出实际决策。

你可以访问：[正在测试的技能]
```

让 Agent 相信这是真实工作，不是测验。

## REFACTOR 阶段：关闭漏洞（保持 Green）

Agent 尽管有技能还是违反了规则？这就像测试回归——你需要重构技能以防止它。

**逐字捕获新的合理化：**
- "这个案例不同，因为..."
- "我遵循精神而非字面"
- "目的是 X，我用不同方式实现 X"
- "务实意味着适应"
- "删除 X 小时是浪费"
- "写测试优先时保留作为参考"
- "我已经手动测试过了"

**记录每个借口。** 这些成为你的合理化表。

### 堵每个漏洞

对于每个新的合理化，添加：

### 1. 规则中的明确否定

<之前>
```markdown
在测试之前写代码？删除它。
```
</之前>

<之后>
```markdown
在测试之前写代码？删除它。重新开始。

**无例外：**
- 不要保留它作为"参考"
- 不要"适应"它同时写测试
- 不要看它
- 删除意味着删除
```
</之后>

### 2. 合理化表中的条目

```markdown
| 借口 | 现实 |
|--------|---------|
| "保留作为参考，先写测试" | 你会适应它。那是之后测试。删除意味着删除。 |
```

### 3. 红旗条目

```markdown
## 红旗——停止

- "保留作为参考"或"适应现有代码"
- "我遵循精神而非字面"
```

### 4. 更新描述

```yaml
description: 当你在测试之前写了代码时，当 tempted 之后测试时，或当手动测试似乎更快时使用。
```

添加即将违反的症状。

### 重构后重新验证

**用更新后的技能重新测试相同场景。**

Agent 现在应该：
- 选择正确选项
- 引用新部分
- 承认他们之前的合理化已被解决

**如果 Agent 找到新的合理化：** 继续 REFACTOR 循环。

**如果 Agent 遵循规则：** 成功——技能对此场景是防弹的。

## 元测试（当 GREEN 不起作用时）

**Agent 选择错误选项后，问：**

```markdown
用户：你读了技能还是选择了选项 C。

该技能如何写得不同才能清楚明白选项 A 是唯一可接受的答案？
```

**三种可能的回答：**

1. **"技能很清楚，我选择忽略它"**
   - 不是文档问题
   - 需要更强的基础原则
   - 添加"违反字面就是违反精神"

2. **"技能应该说 X"**
   - 文档问题
   - 逐字添加他们的建议

3. **"我没看到第 Y 节"**
   - 组织问题
   - 让关键点更突出
   - 尽早添加基础原则

## 技能何时防弹

**防弹技能的迹象：**

1. **Agent 在最大压力下选择正确选项**
2. **Agent 引用技能部分** 作为理由
3. **Agent 承认诱惑** 但无论如何遵循规则
4. **元测试揭示** "技能很清楚，我应该遵循它"

**如果以下情况则非防弹：**
- Agent 找到新的合理化
- Agent 争辩技能是错的
- Agent 创建"混合方法"
- Agent 请求许可但强烈主张违反

## 示例：TDD 技能防弹

### 初始测试（失败）
```markdown
场景：200 行完成，忘了 TDD，疲惫，晚餐计划
Agent 选择：C（之后写测试）
合理化："之后测试达到相同目标"
```

### 迭代 1——添加对策
```markdown
添加部分："为什么顺序重要"
重新测试：Agent 仍然选择 C
新的合理化："精神非字面"
```

### 迭代 2——添加基础原则
```markdown
添加："违反字面就是违反精神"
重新测试：Agent 选择 A（删除它）
引用：直接引用新原则
元测试："技能很清楚，我应该遵循它"
```

**防弹达成。**

## 测试清单（技能的 TDD）

部署技能前，验证你遵循了 RED-GREEN-REFACTOR：

**RED 阶段：**
- [ ] 创建了压力场景（3+ 组合压力）
- [ ] 没有技能运行场景（基线）
- [ ] 逐字记录 Agent 失败和合理化

**GREEN 阶段：**
- [ ] 编写技能解决特定的基线失败
- [ ] 有技能运行场景
- [ ] Agent 现在合规

**REFACTOR 阶段：**
- [ ] 从测试中识别新的合理化
- [ ] 为每个漏洞添加明确对策
- [ ] 更新合理化表
- [ ] 更新红旗列表
- [ ] 用违反症状更新描述
- [ ] 重新测试——Agent 仍然合规
- [ ] 元测试以验证清晰度
- [ ] Agent 在最大压力下遵循规则

## 常见错误（与 TDD 相同）

**❌ 在测试之前编写技能（跳过 RED）**
揭示你认为需要防止什么，不是实际需要防止什么。
✅ 修复：始终先运行基线场景。

**❌ 没有正确观察测试失败**
只运行学术测试，不是真实压力场景。
✅ 修复：使用让 Agent 想要违反的压力场景。

**❌ 弱测试用例（单一压力）**
Agent 抵抗单一压力，在多种压力下崩溃。
✅ 修复：组合 3+ 压力（时间 + 沉没成本 + 疲惫）。

**❌ 没有捕获确切失败**
"Agent 错了"不告诉你需要防止什么。
✅ 修复：逐字记录确切合理化。

**❌ 模糊修复（添加通用对策）**
"不要作弊"不起作用。"不要保留作为参考"起作用。
✅ 修复：为每个特定合理化添加明确否定。

**❌ 第一次通过后停止**
测试通过一次 ≠ 防弹。
✅ 修复：继续 REFACTOR 循环直到没有新的合理化。

## 快速参考（TDD 循环）

| TDD 阶段 | 技能测试 | 成功标准 |
|-----------|---------------|------------------|
| **RED** | 没有技能运行场景 | Agent 失败，记录合理化 |
| **验证 RED** | 捕获确切措辞 | 失败的逐字文档 |
| **GREEN** | 编写技能解决失败 | Agent 现在合规 |
| **验证 GREEN** | 重新测试场景 | Agent 在压力下遵循规则 |
| **REFACTOR** | 关闭漏洞 | 为新的合理化添加对策 |
| **保持 GREEN** | 重新验证 | 重构后 Agent 仍然合规 |

## 底线

**技能创建就是 TDD。相同的原则，相同的循环，相同的收益。**

如果你不会在没有测试的情况下写代码，就不要在没有对 Agent 测试的情况下写技能。

RED-GREEN-REFACTOR 对文档的作用与对代码的作用完全相同。

## 实际影响

将 TDD 应用于 TDD 技能本身（2025-10-03）：
- 6 次 RED-GREEN-REFACTOR 迭代达到防弹
- 基线测试揭示 10+ 独特合理化
- 每次 REFACTOR 关闭特定漏洞
- 最终验证 GREEN：最大压力下 100% 合规
- 相同流程适用于任何纪律执行技能
