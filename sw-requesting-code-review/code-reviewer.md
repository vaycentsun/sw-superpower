## 审查清单

**代码质量：**
- 关注点分离清晰？
- 适当的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 边界情况已处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试真正测试逻辑（而非模拟）？
- 边界情况已覆盖？
- 需要集成测试的地方有集成测试？
- 所有测试通过？

**需求：**
- 所有计划需求已满足？
- 实现与 Spec 匹配？
- 无范围蔓延？
- 破坏性变更已记录？

**生产就绪性：**
- 迁移策略（如 Schema 变更）？
- 考虑向后兼容？
- 文档完整？
- 无显而易见的 Bug？

## 关键规则

**要：**
- 按实际严重程度分类（不是所有都是关键）
- 要具体（文件:行，不要模糊）
- 解释 WHY 问题重要
- 承认优点
- 给出明确结论

**不要：**
- 没有检查就说"看起来不错"
- 把吹毛求疵标记为关键
- 对你没审查的代码给出反馈
- 模糊（"改进错误处理"）
- 避免给出明确结论
